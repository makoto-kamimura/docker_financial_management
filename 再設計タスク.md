## 0. 共通実装規約（全タスク必須）

作業前に必ず以下を守ること。

| 規約 | 内容 |
|---|---|
| API 実装 | 新規・変更する Route Handler は必ず `lib/api-handler.ts` の **`withApi()`** で実装する（`role` / `schema` / `querySchema` を宣言、エラーは `lib/api-error.ts` の `notFound()` / `badRequest()` / `conflict()` / `ApiError` を throw）。`requireRole()` の直接呼び出しは新規に書かない |
| テナント分離 | tenantId 列を持つモデルを schema.prisma に追加したら、`lib/tenant-db.ts` の `TENANT_SCOPED_MODELS` に**必ず登録**する。登録漏れは `src/lib/tenant-db.test.ts` が検出して CI が失敗する |
| 計算ロジック | 集計・計算は `src/lib/*.ts` の**純関数**として実装し、`*.test.ts`（Vitest）を必ず併設する。ルートハンドラには I/O と組み立てだけを置く |
| 会計期間・科目解決 | `lib/period.ts` の `resolvePeriod()` / `findAccountByCode()` / `requireAccountByCode()` を使う（手書きの `period.upsert` を新設しない） |
| マイグレーション | `prisma/migrations/<UTC タイムスタンプ>_<名前>/migration.sql` を手書きで追加（この環境では DB 起動不可のため `prisma migrate dev` は使わない）。既存 SQL の命名規約（`"テーブル名_pkey"` 等）に従う。`npx prisma validate` を通すこと |
| 検証 | `cd app/web && npm run typecheck && npm test && npm run format:check && npm run build` がすべて成功してからコミットする |
| 依存パッケージ | 新規依存の追加は原則禁止（xlsx は削除済み。復活させない）。必要な場合はタスク内で理由を明記 |
| ドキュメント | 実装完了時に `再設計仕様書.md` の該当行を【実装済み】へ更新し、本書のタスクのチェックボックスを埋める |

### F-7【P2・中】元利均等返済の自動計算とオーバーレイ全ローン対応（ステップ 5）✅ 実装済み（number ベース、Decimal 演算は不採用）

- 参照: 仕様書 §6.5【追加設計】、詳細設計書 §9.4
- 変更対象: `src/lib/debt-schedule.ts`（`annuityMonthlyPayment()` / `amortizationSchedule()` 追加）+ `lib/budget-overlay.ts`（`computeHousingLoanOverlay` → 全ローン対応の `computeLoanOverlay` へ拡張・改称、旧名はエイリアスとして残存）+ `/loans` 残高グラフ
- 実装: 月額 = P×r(1+r)^n/((1+r)^n−1)。最終回で丸め誤差清算（Σ元本=P を厳密保証）。`monthlyPayment` 入力済みは入力値優先、未入力は計算値。レスポンスに由来 `source: "manual"|"annuity"` を含める。既存レスポンスキー `housingLoanOverlay` は同一データ（全ローン）で 1 リリース併存。**Decimal 演算は不採用**: 既存 `computeDebtSchedule` が number ベースのため実装方針を統一し、decimal.js 等の新規依存を避けた。`/loans` 残高グラフは今日以降を `amortizationSchedule()` の `remaining` 系列に置換（線形補間を廃止）。ブラウザのタイムゾーンによる月境界ズレを避けるため `Date.UTC` で構築した日付を渡す
- 受け入れ条件: `debt-schedule.test.ts` に r=0 / 1ヶ月 / 最終回清算 / 入力値優先の各ケース — 8 件のテストで実装済み。実サーバー+DB で、`linkedAccountId` 設定済み・`monthlyPayment` 未入力の事業性ローンを作成し `source: "annuity"` での自動計算・住宅ローン（`source: "manual"`）との共存・旧キー `housingLoanOverlay` の後方互換を確認済み。Playwright で残高推移グラフの表示も確認済み

### F-8【P2・小〜中】資産カテゴリ拡張と総資産サマリ（ステップ 4）✅ 実装済み

- 参照: 仕様書 §6.4【追加設計】、詳細設計書 §9.3
- 変更対象: schema（`PersonalAssetCategory` enum に `CASH` / `DEPOSIT` / `SECURITIES` を追加。実体は Prisma enum のため `ALTER TYPE ... ADD VALUE` によるマイグレーションが必要 — 当初「String のため定数追加のみ」としていたのは誤りで、実装時に schema を確認して訂正した）+ 新規 `src/lib/asset-summary.ts`（純関数 `summarizeNetWorth()`）+ 新規 `GET /api/assets/summary` + `/assets` サマリカード（`NetWorthSummaryCard`）
- 実装: 総資産 = 実物資産評価額（`isCountedAsAsset()` フィルタ後）+ 銀行口座残高（`BankTransaction` 明細合計）+ ASSET 科目残高。総負債 = LIABILITY 科目残高 + ローン残高 + 実物資産紐付け負債残高（`computeDebtSchedule()`）。**二重計上の除外規則**を純関数内で実施: ① `linked_accounts.accountId` に紐付く ASSET 科目残高は銀行口座残高側で計上済みのため除外、② `personal_assets.linkedAccountId` に紐付く LIABILITY 科目残高は実物資産紐付け負債側で計上済みのため除外。`LinkedAccount.accountId`（→ `Account.id`）と銀行口座残高側の `BankAccount.id` は別 ID 空間である点に注意して実装。Redis キャッシュ `assets:summary:{tenantId}:{year}-{month}`（1 時間）・`personal-assets` / `loans` / `bank-accounts` の各書き込み API 成功時に `invalidateCache()` で無効化
- 受け入れ条件: `asset-summary.test.ts`（8 件）で除外規則ごとの検証（ASSET 側除外/非除外、LIABILITY 側除外/非除外、`isCountedAsAsset` フィルタ、ローン単独集計、内訳・純資産の総合計算、全ゼロ入力）が成功。実サーバー+DB で、`LinkedAccount` 紐付き ASSET 科目・`PersonalAsset.linkedAccountId` 紐付き LIABILITY 科目それぞれについて curl で除外を確認済み（検証後にテストデータは削除済み）。Playwright で `/assets` の総資産サマリカード表示を確認済み
- 既知の制約: `lib/redis.ts` の `getRedisClient()` はコネクション確立を待たずに `isReady` を同期チェックするため、コールドスタート直後の初回 `invalidateCache()` が無音で no-op になるレース条件が既存する（本タスクで新規に導入したものではなく、キャッシュを使う全機能に共通する既存の制約）。2 回目以降の呼び出しでは正しく動作するため実害は限定的だが、修正するには `lib/redis.ts` の接続処理を async 化し全呼び出し元を見直す必要があり、本タスクのスコープ外とした

> 2026-07-20 に全書き込み・参照経路を棚卸しした結果、当初の前提（「決算書・e-Tax は仕訳を参照」）が**誤り**だと判明した。以下の調査結果を正とし、実装は D-5a〜D-5e のサブタスクに分解した。まず §「D-5 調査結果」を読み、次に依存順（D-5a → D-5b → D-5c → D-5d → D-5e）で着手すること。

#### D-5 調査結果（2026-07-20・コード実測）

**2 系統の実態**

- ①複式仕訳 `JournalEntry`/`JournalDetail` を**参照する集計は 3 つだけ**: 試算表（`api/reports/trial-balance`）・総勘定元帳（`api/reports/general-ledger`）・消費税額控除（`api/tax-credit`）。
- ②簡易実績 `FinancialRecord` を参照するのは**それ以外の全て**: KPI・ダッシュボード・ポータル・キャッシュフロー・予実対比（`lib/budget-actual.ts`）・配分提案・資産サマリ（`assets/summary`）・構成比・予測、**そして決算書（`closing/statements` の P/L・B/S・財務指標）と e-Tax/eLTAX 出力（`closing/etax`）も `FinancialRecord` を読む**。
- つまり乖離は「正式な決算書 vs 非公式ダッシュボード」ではなく、「試算表・総勘定元帳（仕訳ベース・常に貸借一致）vs **決算書・e-Tax 申告データを含むそれ以外全部**（`FinancialRecord` ベース）」。**申告に使う数字が単側書き込みの影響を受ける**ため、`closing/statements`・`closing/etax` は最優先・高影響の consumer として扱う。

**`JournalEntry` を作る経路と同期の欠落**

- `syncJournalToFinancialRecords()` の呼び出しは**全コードベースで 1 箇所のみ**（`api/journals/route.ts:108`）。
- `POST /api/actuals`（`/entry` カレンダータブ。相手科目付きで正しい 2 行仕訳を自動生成する）は**同期を呼ばない** → カレンダー入力の取引は `FinancialRecord` に一切現れない。
- `createSettlementJournal()`（`receivables/[id]/pay`・`payables/[id]/pay` の入金/支払消込。銀行↔AR/AP の完全な複式仕訳を作る）も**同期を呼ばない** → 消込が `FinancialRecord` に届かず、試算表には出るが KPI/ダッシュボード/決算書には出ない。
- `DELETE /api/journals/[id]` は仕訳を消すが、同期で作られた `FinancialRecord` 行を消さない（`FinancialRecord` に `journalEntryId`/`journalDetailId` の**逆参照カラムが無い**ため辿れない）→ 仕訳削除のたびに孤立行が残り恒久的にドリフトする。

**`syncJournalToFinancialRecords()` の実装上の欠陥**（`lib/journal.ts` 全 42 行）

- 呼び出し側が `{ accountId, amount }` だけを渡し、**借方/貸方（`side`）を落としている**。借方行も貸方行も同じ**正の** `amount` で `FinancialRecord` に入る（符号処理なし）。`FinancialRecord` に `side` カラムは無く復元不能。
- P/L 科目（費用の借方・収益の貸方）は「正の額を 1 回計上」が集計側の期待（`closing/statements:65` 等は科目別に `amount` を単純合計）と一致するため**たまたま正しく見える**。しかし B/S 科目を減少側で動かす仕訳（例: `Dr 買掛金 / Cr 現金`）を `POST /api/journals` で登録すると、その科目の `FinancialRecord` 合計に**加算されてしまい残高が膨らむ**（減算されない）。現状はこのバグが顕在化していないのは、唯一の B/S 減少経路である `createSettlementJournal` が同期を呼ばないため（＝潜在バグ）。**同期経路を増やす際は符号規約を先に確定しないと潜在バグを増幅する。**

**単側書き込みを仕訳化できるか（相手科目の情報が揃うか）**

- `/api/financials` 手動・CSV 取込: フォーム/スキーマともに**単科目**（相手科目フィールドが存在しない）→ フォーム拡張 or テナント既定の相手科目（clearing account）の導入が必要。
- 売掛/買掛の発生（`postIssueRecord`）: `Receivable`/`Payable`/`Invoice` に収益/費用科目の FK が無い → 相手科目の導入が必要。
- 銀行明細転記（F-5）: `BankAccount.accountId`（口座に紐付く勘定科目・**任意**）が埋まっていれば `categoryAccountId` と合わせて完全な仕訳を作れる。**単側経路の中で最も仕訳化に近い**が、`accountId` が null の口座では不可。
- 減価償却・棚卸: 対照科目（減価償却累計額・仕入/棚卸調整）がモデル化されておらず情報不足。

#### 目標アーキテクチャ（north star）

`JournalEntry`/`JournalDetail` を「お金が動いた」事実の**唯一の真実**とし、`FinancialRecord` は**単一の choke-point 関数が維持する導出プロジェクション**にする。`FinancialRecord` 行は必ず発生元 `journalEntryId` で辿れる。全書き込み経路は (a) 両サイドの揃った仕訳を作る（→自動でプロジェクションに反映）か、(b) 構造的に単側の経路にはテナント既定の相手/clearing 科目を導入して仕訳化する。到達後、`FinancialRecord` への直接 create は撤廃する。

移行はリスクを抑えるため段階化する（各サブタスクは独立して価値を出せる・独立してレビュー/マージできる粒度）。

##### D-5d-3【手動 `/api/financials`・CSV 取込】検討の結果、対象外と判断（実装しない）

- 検討結果: `/api/financials` は**任意の 1 科目**（B/S・P/L 問わず）を直接入力する手入力/一括取込の経路であり、D-5b で確定したとおり B/S 科目（ASSET/LIABILITY）はスナップショット入力として直接書き込みを維持する設計。P/L 科目の入力についても、相手科目が実在しない（＝どこから/どこへ動いた資金か不明な）入力にテナント既定の「clearing 科目」を割り当てて仕訳化すると、実体のない仕訳（サスペンス科目仕訳）が試算表・総勘定元帳に大量に混入し、かえって元帳の信頼性を落とす。この経路はそもそも「簡易入力（フル複式簿記を書きたくない場合の代替経路）」として存在しており、仕訳化するとその存在意義と矛盾する。
