## 0. 共通実装規約（全タスク必須）

作業前に必ず以下を守ること。

| 規約 | 内容 |
|---|---|
| API 実装 | 新規・変更する Route Handler は必ず `lib/api-handler.ts` の **`withApi()`** で実装する（`role` / `schema` / `querySchema` を宣言、エラーは `lib/api-error.ts` の `notFound()` / `badRequest()` / `conflict()` / `ApiError` を throw）。`requireRole()` の直接呼び出しは新規に書かない |
| テナント分離 | tenantId 列を持つモデルを schema.prisma に追加したら、`lib/tenant-db.ts` の `TENANT_SCOPED_MODELS` に**必ず登録**する。登録漏れは `src/lib/tenant-db.test.ts` が検出して CI が失敗する |
| 計算ロジック | 集計・計算は `src/lib/*.ts` の**純関数**として実装し、`*.test.ts`（Vitest）を必ず併設する。ルートハンドラには I/O と組み立てだけを置く |
| 会計期間・科目解決 | `lib/period.ts` の `resolvePeriod()` / `findAccountByCode()` / `requireAccountByCode()` を使う（手書きの `period.upsert` を新設しない） |
| マイグレーション | `prisma/migrations/<UTC タイムスタンプ>_<名前>/migration.sql` を手書きで追加（この環境では DB 起動不可のため `prisma migrate dev` は使わない）。既存 SQL の命名規約（`"テーブル名_pkey"` 等）に従う。`npx prisma validate` を通すこと |
| 検証 | `cd app/web && npm run typecheck && npm test && npm run format:check && npm run build` がすべて成功してからコミットする |
| 依存パッケージ | 新規依存の追加は原則禁止（xlsx は削除済み。復活させない）。必要な場合はタスク内で理由を明記 |
| ドキュメント | 実装完了時に `再設計仕様書.md` の該当行を【実装済み】へ更新し、本書のタスクのチェックボックスを埋める |

## 2. セキュリティトラック（詳細設計書 §2〜§8）

- 変更対象: `src/app/api/uploads/[filename]/route.ts`、schema（`Receipt.savedName` / `mimeType` 追加 + `fileUrl` からのデータ移行 SQL）
- 実装: **現状は無認証**。`withApi({ role: "viewer" })` 化し、`receipt.findFirst({ where: { savedName, journalEntry: { tenantId } } })` で所有確認（他テナントは 404）。`Content-Disposition: attachment` / `X-Content-Type-Options: nosniff` / `Cache-Control: private, no-store` を付与し、Content-Type は保存時検証値を使う
- 受け入れ条件: 未ログイン 401 / 他テナント receipt 404 のテスト

### S-2【P1・中】MFA の 2 段階化と総当たり対策（詳細設計書 §3.1） ✅ 実装済み

- 変更対象: `src/app/api/auth/login/route.ts`、新規 `POST /api/auth/mfa/verify`、`prisma`（`mfa_challenges` または Redis）
- 実装: `mfa_challenges` テーブル（token は SHA-256 ハッシュ・TTL 5 分）を使う 2 段階フローへ移行。`lib/mfa-challenge.ts` に `issueMfaChallenge()`/`consumeMfaChallenge()` を切り出し、`login/route.ts`（旧互換の password+code 同時送信）と新規 `POST /api/auth/mfa/verify` の両方から共用。MFA 失敗 5 回でチャレンジ失効 + アカウントロック（15分）。`loginAttempts` のリセットはセッション発行時へ移動。`/api/admin/cleanup` に期限切れチャレンジの削除を追加
- 受け入れ条件: mfaToken の TTL・使い捨て・5 回失効 / 旧フロー互換のテスト — `mfa-challenge.integration.test.ts`（実 DB、4 件）+ 実サーバーへの curl で 2 段階フロー・トークン再利用拒否・5 回失敗ロック・旧互換フローを確認済み
- 既知の制約（未実装）: Web `/login`・モバイル `LoginScreen` に MFA コード入力 UI が無いため、MFA 有効ユーザーは現状 API 経由でのみ第 2 段階を完了できる（バックエンドの 2 段階化自体は本タスクで完了）。UI 追加は別タスクとして今後切り出すこと

### S-3【P1・中】セッションのハッシュ保存・二重期限・Bearer ログアウト修正（詳細設計書 §2.3） ✅ 実装済み

- 変更対象: `src/lib/auth.ts`、schema（`Session` に `lastSeenAt` / `ip` / `userAgent`）、`middleware.ts`（Cookie 名定数の共有）
- 実装: トークンは `fm2:` プレフィックス付き乱数を発行し DB には SHA-256 のみ保存（旧平文セッションは 7 日で自然失効）。アイドル 24h + 絶対 7 日。ログイン毎ローテーション。**`destroySession()` を Cookie / Bearer 両対応に修正**（現状モバイルはログアウト不能）。パスワード・ロール変更時の全セッション失効。Cookie は `__Host-` プレフィックス
- 受け入れ条件: 旧→新トークン併存 / Bearer ログアウト / ローテーションのテスト — next/headers の `cookies()`/`headers()` はリクエストスコープ外から呼べないため（実測で確認）、単体テストではなく実サーバー + 実 DB で curl/psql により全項目を確認: `fm2:` トークン発行と DB 側 SHA-256 ハッシュの一致、Bearer ログアウトでの実削除、旧形式（平文）セッションの継続動作、アイドル/絶対期限それぞれ単独での失効、ログイン毎のローテーション（2 セッション同時有効）、ロール変更時の対象セッション失効。既存 F-1 E2E も再確認済み

### S-4【P1・小〜中】CSRF 防御とセキュリティヘッダー（詳細設計書 §4.1・§4.2）✅ 実装済み

- 変更対象: `src/middleware.ts`、`next.config.ts`
- 実装: `/api/*` の POST/PUT/PATCH/DELETE で Origin / Sec-Fetch-Site を検証（Bearer 認証は免除、判定不能な Cookie リクエストは 403）。`headers()` で CSP（まず Report-Only で導入）・HSTS・nosniff・frame-ancestors を全ルートに付与
- 受け入れ条件: middleware 単体テスト（Origin 不一致 403 / Bearer 免除）+ E2E でヘッダー存在確認

### S-5【P2・小】ログイン列挙対策と応答の均一化（詳細設計書 §2.2）

- 変更対象: `src/app/api/auth/login/route.ts`

### S-6【P2・小〜中】パスワードハッシュの非同期化とバージョン付き形式（詳細設計書 §2.1）

### S-8【P2・小〜中】証憑アップロードの検証（詳細設計書 §5.2）

- 依存: S-1
- 変更対象: 新規 `src/lib/upload.ts`、`journals/[id]/receipts/route.ts`
- 実装: サイズ上限 10MB / 拡張子ホワイトリスト（pdf,jpg,jpeg,png,gif,webp）/ マジックバイト検証。保存先を `uploads/<tenantId>/` に分離。証憑削除 API（`DELETE .../receipts/[receiptId]`）追加

### S-9【P2・中】レート制限（詳細設計書 §4.3）

- 変更対象: 新規 `src/lib/rate-limit.ts`、login / mfa / import / sync 各ルート
- 実装: Redis INCR+EXPIRE の固定窓。login は IP 単位 10 回/5 分。超過は 429 + Retry-After。Redis 不通時は認証系のみメモリ内フォールバック

### S-10【P3・小】tenant-db ホワイトリストの DMMF 自動導出（詳細設計書 §6.1）

- 変更対象: `src/lib/tenant-db.ts`
- 実装: 手書き Set を `Prisma.dmmf` から「tenantId 保有モデル − EXCLUDED」で起動時構築へ置換（回帰テストは実装済みなので挙動同値を確認するだけでよい）

### S-11【P3・中】CSV 取込の堅牢化（詳細設計書 §7）

- 変更対象: `src/lib/import.ts`、`src/lib/banktxn-import.ts`
- 実装: 行数上限 10,000 / ファイル 5MB / 科目・期間の一括解決（N+1 解消）/ `$transaction` で全体原子化（エラー行があれば全体中止に**仕様変更**。リリースノート記載）

### S-12【P3・小〜中】監査ログ強化（詳細設計書 §8）

- 変更対象: `src/lib/audit.ts`、schema（`AuditLog` に `tenantId` / `ip` / `userAgent`）、`lib/api-handler.ts`（`ctx.audit` への自動付与）
- 実装: `redactSecrets()` で passwordHash / totpSecret / mfaRecoveryCodes をマスク。書き込み失敗は `console.error` で観測可能に

### S-13【P3・小】環境変数の起動時検証とサービスキー定数時間比較（詳細設計書 §1.3・§4.4）

- 変更対象: 新規 `src/lib/config.ts`、`admin/cleanup/route.ts`
- 実装: Zod で環境変数を起動時検証（`CLEANUP_SERVICE_KEY` は 32 文字未満拒否）。cleanup のキー比較を `timingSafeEqual` 化

### S-14【P2・小】モバイルセッションの SecureStore 永続化（詳細設計書 §2.4）

- 変更対象: `app/mobile/src/api.ts`、`App.tsx`（依存追加: `expo-secure-store` — 本タスクのみ依存追加を許可）
- 実装: セッショントークンを SecureStore に保存し、起動時に `GET /api/auth/me` で有効性確認。AsyncStorage は使わない

### S-15【P2・小〜中】MFA コード入力 UI（Web / モバイル）

- 参照: S-2（バックエンドの 2 段階化は実装済み）
- 変更対象: `src/app/login/page.tsx`（新規: MFA コード入力ステップ）、モバイル `app/mobile/src/screens/LoginScreen.tsx` + `api.ts`（新規 `verifyMfa()`）
- 実装: `POST /api/auth/login` が `{ mfaRequired: true, mfaToken }` を返した場合に、TOTP コード（6桁）またはリカバリーコード入力欄を表示し `POST /api/auth/mfa/verify` を呼ぶ 2 画面目を追加する。ロック時（429）のメッセージ表示も含める
- 受け入れ条件: MFA 有効ユーザーが Web / モバイルの UI 操作のみでログインを完了できる E2E（またはモバイルは手動確認）
