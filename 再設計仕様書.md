## 1. システムコンセプト

**「家計簿をつける感覚で始めて、法人会計の考え方まで学べる」段階的ステップアップ型の統合会計管理システム。**

ユーザーはまず家計簿として収入・支出を記録することから始め、予算配分 → 資金フローの可視化 → 総資産の把握 → ローンを考慮した予算計画へと管理レベルを段階的に引き上げる。最終ステップでは、それまで蓄積した家計の勘定項目を法人会計の科目表記に置き換えて表示し、法人の決算処理に必要な考え方・処理・申告書類を**自分のデータで**学べる。

| 項目 | 内容 |
|---|---|
| 想定ユーザー | 家計管理担当者（入口）→ 個人事業主・中小法人の経営者・経理担当・税理士（出口） |
| 提供形態 | Web アプリ（PC ブラウザ）＋ モバイルアプリ（iOS / Android） |
| バックエンド | Next.js App Router Route Handlers（`/api/*`） |
| DB | PostgreSQL 16（Prisma ORM） |
| キャッシュ | Redis 7（TTL 1 時間、接続失敗時は DB フォールスルー） |
| マルチテナント | `tenants` 単位で全財務データを分離（1 テナント : N ユーザー） |
| 設計原則 | 「一家庭 = 一法人」モデル。家計の科目体系（`H` prefix）を法人会計と同一のデータ基盤（勘定科目・予算・実績・複式仕訳）の上に置くことで、ステップ 6 の表記切り替え・学習を無理なく成立させる |

## 2. 利用ステップの定義（全体像）

システムの利用は以下の 6 ステップとして定義する。各ステップは前のステップの入力データを土台とし、**登録するもの**と**確認できるようになるもの（ゴール）** が明確に対応する。

| # | ステップ | 登録するもの | 確認できるようになるもの（ゴール） | 実装状況 |
|---|---|---|---|---|
| 1 | 家計簿として記録する | 予算（各収入）・実績（各支払項目） | ダッシュボードで**収支にマイナスがないか** | ✅ 実装済み（マイナス警告表示を含む） |
| 2 | 収入から予算を配分する | 提示された配分率（%）＝金額をもとにした正式な予算（各支払項目） | ダッシュボードで**支払項目ごとの予実差** | ✅ 実装済み |
| 3 | お金の流れを可視化する | 銀行口座・クレジットカード情報、入出金履歴、収入・支払項目との紐付け | **月間の口座間・カード間のお金の流れ（キャッシュフロー図）** | 🔶 一部実装（明細と科目の紐付け・転記は実装済み。実績ベース月間フロー図は追加設計） |
| 4 | 総資産を把握する | 現金（タンス預金）・預金・投資額・不動産・車などの現時点評価額 | **総資産（純資産）** | 🔶 一部実装（現金・投資カテゴリと総資産サマリが追加設計） |
| 5 | ローンを予算に織り込む | ローン・借入金の金額・登録時利率・支払開始日・支払終了日 | 支払期間中は**割賦金額が予算へ自動計上**され、それを考慮した予算配分 | ✅ 実装済み |
| 6 | 法人会計を学ぶ | —（表記の切り替え操作） | 勘定項目を**法人の決算処理に必要な科目表記へ置き換えた表示**。法人決算の考え方・処理・必要申告書を学べる | 🔶 一部実装（表記切替の画面反映は実装済み・学習コンテンツが追加設計） |

### ステップ間のデータ依存関係

```
STEP 1  予算(収入) ── 実績(支払項目) ──────────────┐
           │                │                      │
STEP 2  収入合計 × 配分% → 予算(支払項目) → 予実差 │
           │                │                      ▼
STEP 3  口座・カード ← 明細の紐付け ←──── 実績  フロー図
           │
STEP 4  口座残高 + 実物資産評価額 → 総資産
           │
STEP 5  ローン（利率・期間）→ 割賦金額 → 予算へ自動計上（STEP 2 の配分に反映）
           │
STEP 6  蓄積された勘定項目 → 法人科目表記へ変換・学習
```

- ステップ 1〜5 は**家計モード（`household`）** で完結する。
- ステップ 6 で**個人事業主モード（`sole`）／法人モード（`corporate`）** へ切り替える。
- 各ステップは順不同でも利用できるが、上記の順で進めると入力済みデータが次のステップの初期値・提案に活きる。

## 5. 観点モードと利用ステップの関係

同一データ基盤の上で表示ラベル・ナビゲーション・利用機能を切り替える 3 つの観点モードを持つ。利用ステップとの対応は以下のとおり。

| モード | viewMode キー | 対応ステップ | システム名 |
|---|---|---|---|
| 家計 | `household` | **ステップ 1〜5**（日常利用の主戦場） | 家計管理システム |
| 個人事業主 | `sole` | **ステップ 6**（複式簿記・確定申告の学習と実務） | 個人会計システム |
| 法人 | `corporate` | **ステップ 6**（法人決算・ガバナンスの学習と実務） | 法人会計システム |

- **Web**: `AppShell.tsx` のサイドバー切替ピル。localStorage（キー `viewMode`）に永続化。初期値 `sole`（→ ステップ導線を重視するなら `household` 初期化が望ましい。付録 A 参照）。
- **モバイル**: ヘッダーの 3 ボタンピル。メモリ内保持、初期値 `sole`。

### 5.1 モード別ラベル対応（資金フロー・KPI）

| フィールド | household | sole | corporate |
|---|---|---|---|
| `revenue` | 収入 | 売上 | 売上高 |
| `cogs` | 変動費 | 仕入・変動費 | 売上原価 |
| `grossProfit` | 収支差額 | 粗利 | 売上総利益 |
| `expense` | 固定費 | 経費 | 販管費 |
| `operatingProfit` | 手残り | 事業利益 | 営業利益 |

### 5.2 勘定科目カテゴリ（`AccountCategory`）とモード別解釈

| カテゴリ | household 表示 | sole 表示 | corporate 表示 |
|---|---|---|---|
| `REVENUE` | 収入（給与・副業・配当） | 売上（売上高・雑収入） | 売上高（商品売上・役務収益） |
| `COGS` | 変動費（食費・日用品・交通費） | 仕入・変動費（仕入・外注費） | 売上原価（仕入高・製造原価） |
| `EXPENSE` | 固定費（家賃・保険・通信費） | 経費（通信費・地代家賃） | 販管費（人件費・広告費） |
| `PROFIT` | 貯蓄/利益 | 損益（事業所得・青色控除） | 損益（営業・経常・当期純利益） |
| `ASSET` | 資産（預金・有価証券・不動産） | 資産（現金・売掛金・固定資産） | 資産（現預金・繰延税金資産） |
| `LIABILITY` | 負債（住宅ローン等） | 負債（買掛金・借入金） | 負債（未払法人税・社債） |

### 5.3 科目コード規約と既定科目

- 家計: `H` prefix（例: H1000 給与・賞与 / H2000 食費 / H3000 家賃・住宅ローン / H9000 貯蓄・投資積立）。テナント作成時に**既定 76 科目**を自動投入（`lib/default-accounts.ts`）。
- 個人事業主: 数字始まり。法人: `C` prefix（モバイルの科目フィルタ `matchesViewMode()` が準拠）。
- 各科目はモード別表示名（`Account.soleName` / `Account.corporateName`）を保持し、null は家庭科目名 `name` にフォールバック（ステップ 6 の基盤）。

## 6. ステップ別機能仕様

各ステップについて「ユーザー操作フロー」「使う画面・API」「実装状況」「追加設計」を定義する。**【実装済み】** は現行コードで動作するもの、**【追加設計】** は本再設計で新規定義するものを指す。

---

### 6.1 ステップ 1: 家計簿として記録する

> 家庭の家計簿をつける感覚で、予算（各収入）を登録し、実績（各支払項目）を登録する。
> **ゴール: ダッシュボードで収支にマイナスがないか確認できる。**

#### 操作フロー

1. ログイン → 家計モードを選択（既定科目 76 件が投入済み）
2. `/budget` で**収入科目**（H1000 給与・賞与など）に月次予算を登録
3. `/entry` で日々の**支払項目**（H2000 食費など）の実績を登録
   - 明細フォーム / カレンダー入力（日付クリック → 支出/収入トグル・科目・金額・支払方法）/ CSV インポート（`accountCode,fiscalYear,month,amount`）
4. `/dashboard` で当月の収入・支出・貯蓄額（= 手残り）を確認

#### 使う画面・API

| 機能 | 画面 / API | 状況 |
|---|---|---|
| 収入予算の登録（科目×月マトリクス・セル直接編集） | `/budget` / `GET POST /api/budgets`、`PATCH DELETE /api/budgets/[id]`、`POST /api/budgets/import` | 【実装済み】 |
| 実績の明細登録・履歴 | `/entry` / `GET POST /api/financials`、`GET /api/financials/recent`（直近 30 件）、`GET /api/financials/[id]/history` | 【実装済み】 |
| カレンダー入力（日次。内部的に複式仕訳を自動生成） | `/entry` カレンダータブ / `GET POST DELETE /api/actuals` | 【実装済み】 |
| CSV 一括取込 | `POST /api/financials/import` | 【実装済み】（Excel/xlsx 対応は 2026-07-13 に廃止。依存パッケージごと削除） |
| 収支確認 KPI（収入 / 支出 / 貯蓄額・貯蓄率 / YTD / MoM / YoY） | `/dashboard` / `GET /api/kpi` | 【実装済み】 |
| 月次推移・予実対比グラフ | `/dashboard` 予実対比タブ / `GET /api/reports/budget-actual` | 【実装済み】 |

#### 収支マイナス警告 【実装済み】

「マイナスがないか確認できる」を明示化するため、ダッシュボードに以下を追加する。

| 項目 | 内容 |
|---|---|
| 当月赤字警告 | KPI「貯蓄額」カードで `operatingProfit < 0` のとき赤色強調 + 警告アイコン・メッセージ（「今月は支出が収入を上回っています」） |
| 赤字月ハイライト | 予実対比の月次テーブル・グラフで、実績支出合計 > 実績収入合計 の月を赤背景表示 |
| 実装方式 | 既存 `GET /api/kpi` / `GET /api/reports/budget-actual` のレスポンスで判定可能なため**画面側のみの変更**（API 追加不要） |

### 6.2 ステップ 2: 収入から予算を配分する

> 収入金額から、各支払項目に予算配分すべきパーセンテージ＝金額を提示し、正式に予算（各支払項目）を登録する。
> **ゴール: ダッシュボードで支払項目ごとの予実差を確認できる。**

#### 操作フロー

1. ステップ 1 で登録した収入（予算または実績）の月合計を基準額とする
2. システムが**配分ルール**（支払項目ごとの推奨 %）から科目別の推奨予算額（最小〜最大）を提示
3. ユーザーが金額を調整して**一括で予算登録**
4. `/dashboard`・`/reports` で支払項目ごとの予算 vs 実績（差異・達成率）を確認

#### 配分ルール（推奨パーセンテージ）

#### 利率を考慮した割賦金額の自動計算と適用範囲の拡大 【実装済み】

| 項目 | 内容 |
|---|---|
| 元利均等返済の自動計算 | `lib/debt-schedule.ts` の `annuityMonthlyPayment()`: 月額 = P × r(1+r)^n / ((1+r)^n − 1)（P=借入額、r=月利=年利/12、n=支払開始日〜終了日の月数）。`monthlyPayment` 未入力時はこの計算値を割賦金額として採用し、入力済みなら入力値を優先。number ベースで実装（既存 `computeDebtSchedule` と方針統一。decimal.js 等の新規依存は不採用） |

## 10. マルチテナント方式（データ分離）

`tenants` を財務データの分離単位とする。**1 テナントに複数ユーザーを所属させられる（Users : Tenant = N : 1）**。

| 項目 | 内容 |
|---|---|
| モデル | `users.tenantId Int`（非ユニーク、index 付き）。`POST /api/admin/users` は既定で作成者と同じテナントへ追加し、`newTenant: true` 指定時のみ空の新規テナントをトランザクションで自動作成 |
| ユーザー管理の分離 | `/api/admin/users` は自テナントのユーザーのみ対象。他テナントは一覧非表示・ID 指定操作は 404 |
| API 側の強制 | 全 API ルートで `lib/tenant-db.ts` の `tenantDb(tenantId)` によりテナントスコープを強制。他テナントのリソースへの ID 指定アクセスは 403 / 404 |
| 分離対象 | accounts / periods / budgets / financial_records / departments / bank_accounts / transfers / journal_entries / journal_templates / invoices / loans / receivables / payables / fixed_assets / inventories / apportionments / linked_accounts / business_profiles / tax_settings / fiscal_years / fiscal_year_closes / officers / shareholder_meetings / dividends / announcements / accrued_revenues / accrued_expenses / personal_assets / forecasts（★追加テーブルも同様に tenantId を付与） |
| 複合ユニークキー | 単体ユニーク制約を `tenantId` を含む複合キーに変更（§9 の各 `@@unique` 参照） |

| 科目自動投入 | テナント作成時に家庭モード既定科目（モード別表示名込み）を `seedDefaultAccountsForTenant()` で自動登録。配分ルール既定値も `seedDefaultAllocationRulesForTenant()` で同時投入（実装済み） |

## 11. 認証・セキュリティ

### 11.1 認証（`lib/auth.ts`）

- パスワードは **scrypt**（salt 16 バイト + 派生鍵 64 バイト、`salt:hash` 形式）でハッシュ化。外部ライブラリ依存なし。
- セッションは **DB 保存**（`sessions`、ID は 32 バイトランダム hex）。**有効期間 7 日**。
- **Web**: httpOnly Cookie `fm_session`（`sameSite: lax`、本番は `secure`。`COOKIE_SECURE=false` で無効化可）。
- **モバイル**: ログインレスポンスのセッション ID を `Authorization: Bearer <sessionId>` で送信（JWT は不使用）。
- **画面ガード**: `src/middleware.ts` が保護対象パスで Cookie の有無をチェックし `/login` へリダイレクト。セッション有効性は各 API 側で検証。
- **アカウントロック**: 連続 **5 回**失敗で **15 分間**ロック。成功時にカウンタリセット。
- **セッション掃除**: Compose の `session-cleanup` コンテナが毎日 02:00 に `POST /api/admin/cleanup` を `CLEANUP_SERVICE_KEY` 認証で実行。

### 11.2 RBAC（4 ロール、`lib/authz.ts` の `requireRole()`）

| ロール | ランク | 主な権限 |
|---|---|---|
| viewer | 1 | 全画面・全 API の閲覧のみ |
| accountant | 2 | viewer + 仕訳承認ワークフロー・e-Tax XML 生成・税理士ポータル |
| editor | 3 | accountant + データ登録・編集・削除・決算確定/解除 |
| admin | 4 | editor + ユーザー管理・監査ログ・テナント作成/削除・会計年度削除・クリーンアップ |

### 11.3 MFA（TOTP、`lib/totp.ts`）

- RFC 6238 TOTP（SHA1 / 30 秒 / 6 桁）、外部ライブラリ依存なし。
- `/settings` でシークレット発行（otpauth URI）→ 認証アプリ登録 → 6 桁コードで有効化 → 以降ログイン時に必須。
- リカバリーコード: SHA-256 ハッシュを `users.mfaRecoveryCodes` に JSON 保存。TOTP の代わりに使用可能（使用済みは無効化）。

### 11.4 監査ログ（`lib/audit.ts`）

ログイン・データ変更を `audit_logs` に before/after JSON 差分付きで記録。`/admin/audit`（admin）で閲覧。

### 11.5 【追加設計】セキュリティ再設計（詳細は `再設計詳細設計書.md`）

| 優先 | 項目 | 現状の課題 | 変更後（詳細設計の節） |

### 15.2 CD（`.github/workflows/cd.yml`、main push / `v*` タグ / 手動）

1. `build-and-push`: GHCR（`ghcr.io/<repo>/web`）へイメージ push（タグ: ブランチ名 / semver / SHA）。
2. `deploy`（environment: production）: compose を scp 配置 → SSH で `pull` → `up -d` → `prisma migrate deploy` → `/api/health` を最大 60 秒リトライ。**失敗時は直前イメージへ自動ロールバック**（DB マイグレーションは戻さない）→ `exit 1`。成功時のみ `docker image prune -f`。`HEALTHCHECK_URL` 外形チェック・`MAIL_TO` メール通知は任意。

| `DEPLOY_HOST` / `DEPLOY_USER` / `DEPLOY_SSH_KEY` / `DEPLOY_PATH` | SSH デプロイ | ✅ |
| `DEPLOY_PORT` / `GHCR_USER` / `GHCR_PAT` / `HEALTHCHECK_URL` | ポート / GHCR private / 外形チェック | 任意 |
| `MAIL_SERVER` / `MAIL_PORT` / `MAIL_USERNAME` / `MAIL_PASSWORD` / `MAIL_FROM` / `MAIL_TO` | メール通知 | 通知時 |
| `GITHUB_TOKEN` | GHCR push（自動付与） | 自動 |
