## 0. 設計方針

### 0.1 セキュリティ原則

| 原則 | 適用 |
|---|---|
| 多層防御 | middleware（Origin 検証・ヘッダー付与）→ ルート（認証・認可・入力検証）→ lib（テナントスコープ強制）→ DB（複合ユニーク制約）の各層で独立に防御する |
| 秘密情報は導出値で保存 | パスワード=scrypt、セッション ID=SHA-256、リカバリーコード=SHA-256。**DB が漏えいしてもそのまま成りすましに使えない**状態を保つ |
| フェイルセキュア | 判定不能・例外時は拒否側に倒す（例: Origin 不明の状態変更リクエストは 403） |
| 応答の均一化 | 認証系の失敗応答は存在有無・原因によらず同一メッセージ・同等の処理時間にする |
| 監査可能性 | 認証イベント・データ変更・機械計上（オーバーレイ）を監査ログで追跡できる。予算の自動計上は既存どおり**オーバーレイ方式**（Budget テーブル非破壊）を維持する |

### 0.2 保守性原則

| 原則 | 適用 |
|---|---|
| ボイラープレートの共通化 | 全ルートで繰り返される「認証 → 認可 → 入力検証 → テナント DB → エラー整形 → 監査」を `withApi()` に集約（§1.1） |
| 純関数のドメインロジック + 単体テスト | 計算ロジック（元利均等・配分提案・総資産集計）は `lib/*.ts` の純関数として実装し、既存の `lib/*.test.ts` パターンで Vitest テストを併設する |
| 「仕組み」による再発防止 | 手動チェックリスト（tenant-db のホワイトリスト等）は、スキーマから自動導出 + 回帰テストへ置き換える（§6） |
| 設定の起動時検証 | 環境変数は `lib/config.ts` で Zod により起動時に検証し、実行時の `process.env` 直参照を排除する（§1.3） |
| 段階的移行 | ハッシュ形式・セッション形式はバージョン識別子を持たせ、旧形式と併存させたまま無停止で移行する（§10） |

## 1. 共通基盤の再設計（保守性）

### 1.1 API ハンドラ共通化: `lib/api-handler.ts`（新規）

**【現状】** 約 70 ファイルの `route.ts` それぞれで `requireRole()` 呼び出し、`Number(id)` 変換、Zod parse、try/catch、`NextResponse.json` 整形を手書きしている（例: `app/api/journals/[id]/receipts/route.ts`）。バリデーション漏れ・NaN 混入（`Number("abc")`）・エラー形式の不統一が構造的に起こりうる。

**【変更後】** ルート定義を宣言的にする高階関数を新設する。

```ts
// lib/api-handler.ts
type ApiContext<TBody, TQuery> = {
  user: AuthUser;            // requireRole 済みユーザー（tenantId 含む）
  db: TenantDb;              // tenantDb(user.tenantId) 済みクライアント
  body: TBody;               // Zod 検証済みボディ（schema 指定時）
  query: TQuery;             // Zod 検証済みクエリ（querySchema 指定時）
  params: Record<string, string>;
  id?: number;               // [id] ルートの場合、正の整数として検証済み
  audit: (action: string, target: string, opts?: AuditOptions) => Promise<void>;
};

export function withApi<TBody = never, TQuery = never>(opts: {
  role: Role;                          // 最低ロール（既存 requireRole を内部利用）
  schema?: ZodSchema<TBody>;           // ボディ検証
  querySchema?: ZodSchema<TQuery>;     // クエリ検証
  handler: (ctx: ApiContext<TBody, TQuery>) => Promise<Response>;
}): (req: NextRequest, route: { params: Promise<Record<string, string>> }) => Promise<Response>;
```

処理順序（すべて `withApi` 内部で実施）:

1. `requireRole(opts.role)` — 401 / 403（既存 `lib/authz.ts` を内部で使用）
2. `[id]` パラメータの検証 — `z.coerce.number().int().positive()`。不正なら 400（現状の `Number(id)` → NaN で Prisma に渡る経路を遮断）
3. ボディ / クエリの Zod 検証 — 失敗時は 400。**レスポンスにはフィールド名と定型メッセージのみ**を含め、Zod の内部構造（`error.flatten()` 全体）は返さない
4. `tenantDb(user.tenantId)` を生成し ctx へ
5. `handler` 実行。例外は §1.2 のエラーマッパーで整形
6. `ctx.audit()` は userId・tenantId・リクエスト IP を自動付与した `writeAudit` ラッパー（§8）

**適用方針**: 新規 API（§9）は必ず `withApi` で実装する。既存ルートはセキュリティ修正（§2〜§5）と同時に順次移行し、`requireRole` 直呼びの新規追加を ESLint ルール（`no-restricted-imports` で `authz` の直 import をルート層のみ警告）で抑止する。

### 1.2 エラーレスポンスの標準化: `lib/api-error.ts`（新規）

**【現状】** `{ error: "not found" }` / `{ error: parsed.error.flatten() }` / 日本語文字列など形式が混在。未捕捉例外時に Next.js 既定の 500 が返る。

**【変更後】**

```ts
// 全 API 共通のエラー封筒
type ApiError = { error: { code: string; message: string; fields?: Record<string, string> } };
```

| 状況 | code | HTTP | 備考 |
|---|---|---|---|
| 未ログイン | `unauthorized` | 401 | |
| 権限不足 | `forbidden` | 403 | |
| 入力不正 | `validation_error` | 400 | `fields` にフィールド別定型メッセージ |
| 対象なし・他テナント | `not_found` | 404 | **他テナント資源の存在は 404 で秘匿**（403 を返さない） |
| 一意制約違反 | `conflict` | 409 | Prisma `P2002` を写像 |
| レート制限 | `rate_limited` | 429 | `Retry-After` ヘッダー付き |
| 内部エラー | `internal_error` | 500 | メッセージ固定。スタック・SQL・Prisma エラー詳細はログのみに出力し**レスポンスへ含めない** |

Prisma 例外 → HTTP の写像（`P2002`→409、`P2025`→404）を `withApi` の catch 節で一元化する。

### 1.3 環境変数の起動時検証: `lib/config.ts`（新規）

**【現状】** `process.env.CLEANUP_SERVICE_KEY`・`COOKIE_SECURE` 等を各所で直接参照。未設定・誤設定が実行時まで発覚しない。

**【変更後】** Zod スキーマで起動時に一括検証し、型付きオブジェクトとして export する。

```ts
const Env = z.object({
  DATABASE_URL: z.string().url(),
  REDIS_URL: z.string().url().optional(),
  CLEANUP_SERVICE_KEY: z.string().min(32).optional(),   // 短い鍵を拒否
  COOKIE_SECURE: z.enum(["true", "false"]).default("true"),
  SESSION_IDLE_HOURS: z.coerce.number().default(24),
  SESSION_ABSOLUTE_DAYS: z.coerce.number().default(7),
  UPLOAD_MAX_BYTES: z.coerce.number().default(10 * 1024 * 1024),
  APP_ORIGIN: z.string().url().optional(),               // CSRF Origin 検証（§4.1）
});
export const config = Env.parse(process.env);  // 不正なら起動失敗（フェイルファスト）
```

`CLEANUP_SERVICE_KEY` は 32 文字未満を拒否し、`docs/deploy.md` 記載の例のような推測可能な値の本番混入を防ぐ。

## 2. 認証ロジックの再設計

対象: `lib/auth.ts`・`app/api/auth/login/route.ts`・`app/api/auth/logout/route.ts`

### 2.1 パスワードハッシュ: 非同期化・パラメータ明示・バージョン付き形式

**【現状】** `auth.ts:9-22` — `scryptSync`（同期）で N 等のコストパラメータは Node 既定値。形式は `salt:hash` でパラメータが埋め込まれておらず、将来の強度変更時に判別・移行ができない。同期実行はイベントループを塞ぐため、ログイン試行の並列送信だけで全リクエストが遅延する（DoS ベクタ）。

**【変更後】**

```
形式:  scrypt$<N>$<r>$<p>$<salt(base64)>$<hash(base64)>
既定:  N=2^15 (32768), r=8, p=1, keyLen=64, salt=16byte
```

- `crypto.scrypt`（**非同期**）+ `util.promisify` に置換。`hashPassword` / `verifyPassword` は `Promise` を返すシグネチャに変更（呼び出し元は login / MFA / admin ユーザー作成の 3 箇所）。
- 検証時はハッシュ文字列からパラメータを読み取って照合する。**旧形式 `salt:hash` も判別して検証可能**とし、旧形式で検証成功したログイン時に新形式で再ハッシュして保存する（透過移行、§10）。
- 比較は既存どおり `timingSafeEqual`。
- パスワードポリシー: 新規設定・変更時は Zod で最小 12 文字を強制（seed の `password` は開発用のみ。§10 で本番 seed 分離）。

### 2.2 ユーザー列挙対策: 応答時間の均一化

**【現状】** `login/route.ts` — ユーザー不在時は scrypt を実行せず即座に 401 を返す。コメントは「同一タイミング応答で列挙を防ぐ」と書かれているが、実際には**ハッシュ計算の有無で応答時間に明確な差が出る**ため、メールアドレスの在籍が推測できる。

**【変更後】** ユーザー不在時も**ダミーハッシュに対して必ず `verifyPassword` を実行**してから同一メッセージの 401 を返す。

```ts
// モジュール定数（起動時に一度生成）
const DUMMY_HASH = await hashPassword(randomBytes(32).toString("hex"));

const user = await prisma.user.findUnique({ where: { email } });
const ok = await verifyPassword(password, user?.passwordHash ?? DUMMY_HASH);
if (!user || !ok) { /* 共通の失敗処理へ */ }
```

失敗メッセージも統一する。現状の「残り N 回でロック」はロック閾値を攻撃者に開示するため、**「メールアドレスまたはパスワードが正しくありません」に固定**し、残回数はレスポンスに含めない（監査ログには記録する）。

### 2.3 セッション管理: ハッシュ保存・ローテーション・二段階の期限

**【現状】** `auth.ts:25-71` —
1. セッション ID（32 バイト hex）を**平文のまま** `sessions.id` に保存。DB ダンプが漏えいすると全セッションが即座に乗っ取り可能。
2. 有効期限は発行時から固定 7 日のみ（アイドルタイムアウトなし）。
3. `destroySession()` が Cookie しか参照しないため、**モバイル（Bearer）からの `POST /api/auth/logout` ではセッションが削除されない**（実質ログアウト不能バグ）。
4. ロール変更・パスワード変更時に既存セッションが残り続ける。

**【変更後】**

#### スキーマ変更

```prisma
model Session {
  id          String   @id            // ← SHA-256(token) の hex を保存（平文トークンは保存しない）
  userId      Int
  createdAt   DateTime @default(now())
  lastSeenAt  DateTime @default(now()) // アイドルタイムアウト判定
  expiresAt   DateTime                 // 絶対期限（発行から 7 日）
  ip          String?                  // 監査用
  userAgent   String?                  // 監査用
  user        User     @relation(...)
  @@index([userId])
  @@index([expiresAt])
}
```

#### トークンライフサイクル

| 項目 | 設計 |
|---|---|
| 発行 | `token = randomBytes(32).toString("hex")` をクライアントに渡し、DB には `sha256(token)` のみ保存。旧セッション（平文 ID）とは**桁数が同じため、`fm2:` プレフィックスをトークンに付けて**新旧を判別する（§10） |
| 検証 | `lastSeenAt + SESSION_IDLE_HOURS(24h) > now` かつ `expiresAt(7日) > now` の両方を要求。検証成功時、`lastSeenAt` が 1 時間以上古い場合のみ更新（書き込み頻度の抑制） |
| ログアウト | `destroySession()` を **Cookie と Bearer の両方**から解決するよう修正（`getCurrentUser` と同じ解決ロジックを共通関数 `resolveSessionToken(req)` に抽出）。これで現状のモバイルログアウト不能を解消 |
| 全端末ログアウト | パスワード変更・ロール変更・MFA 設定変更時に `session.deleteMany({ where: { userId } })`（操作中の自セッションは再発行）。`PATCH /api/admin/users/[id]` のロール変更にも適用 |
| ローテーション | ログイン成功時は常に新規トークン発行（セッション固定攻撃対策。Cookie 上書き） |

#### Cookie 属性

```ts
store.set("__Host-fm_session", token, {
  httpOnly: true,
  secure: config.COOKIE_SECURE === "true",  // 既定 true（現状の「本番のみ secure」から反転）
  sameSite: "lax",
  path: "/",
  expires: absoluteExpiry,
});
```

- `__Host-` プレフィックスで secure・path=/・domain 未指定を強制する（`COOKIE_SECURE=false` の開発時のみ従来名 `fm_session` にフォールバック。プレフィックスは secure 必須のため）。
- `middleware.ts` の Cookie 名参照も定数を共有する（`lib/session-constants.ts` に切り出し、Edge ランタイムから import 可能にする）。

### 2.4 モバイルのセッション保持

**【現状】** セッション ID をメモリ内モジュール変数に保持（再起動で消滅）。仕様書 §12 の既知の制約。

**【変更後】** `expo-secure-store`（iOS Keychain / Android Keystore）に保存する。AsyncStorage は平文保存のため**使用しない**。API は既存の `Authorization: Bearer <token>` を維持（サーバー側は §2.3 のハッシュ照合に変わるがクライアントには透過）。起動時に SecureStore から復元 → `GET /api/auth/me` で有効性確認 → 無効なら破棄してログイン画面へ。

## 3. MFA（TOTP）の再設計

対象: `lib/totp.ts`・`app/api/auth/login/route.ts`・`app/api/auth/mfa/*`

### 3.1 ログインフローの 2 段階化（MFA 総当たり対策）

**【現状】** `login/route.ts` は次の順で処理している:

1. パスワード検証に成功すると**その時点で `loginAttempts` を 0 にリセット**
2. その後に TOTP を検証するが、**TOTP 失敗は失敗カウントに数えられずロックも発動しない**

このため、パスワードを知る攻撃者（またはフィッシング後の攻撃者）が 6 桁 TOTP（100 万通り、±1 ステップ許容で実質 3 窓）を**無制限に総当たりできる**。また毎回のリクエストでパスワードが再送される。

**【変更後】** パスワード認証と MFA 認証を別トークンで分離する。

```
POST /api/auth/login    { email, password }
  → MFA 無効: セッション発行（従来どおり）
  → MFA 有効: { mfaRequired: true, mfaToken } を返す（セッションは発行しない）

POST /api/auth/mfa/verify   { mfaToken, code | recoveryCode }   ← 新設
  → 成功: セッション発行
  → 失敗: mfa_attempts をインクリメント。5 回失敗で mfaToken 失効 + アカウントロック（§2 と同じ 15 分）
```

- `mfaToken`: `randomBytes(32)` の使い捨てトークン。Redis（`mfa:pending:<sha256(token)>` → userId、TTL 5 分、フォールバック時は `sessions` テーブルに `kind` 列を持たせず専用テーブル `mfa_challenges` を使用）に保存。**検証成功・失敗上限・TTL のいずれかで必ず消滅**する。
- `loginAttempts` のリセットは**セッション発行時**（=MFA も通過した後）に移動する。パスワード成功のみではリセットしない。
- 旧クライアント互換: 既存 Web / モバイルの「password + code 同時送信」も 1 リリースの間は受け付け（内部で 2 段階を連続実行）、その間に画面を新フローへ移行する（§10）。

### 3.2 TOTP リプレイ防止

**【現状】** `totp.ts` は ±1 ステップの照合のみで、**同一コードの使い回しを検出しない**。盗聴・ショルダーハックで得たコードが 30〜90 秒間再利用できる。

**【変更後】** `users` に `totpLastUsedStep BigInt?` を追加。検証成功時に使用ステップを記録し、**記録値以下のステップのコードは拒否**する。

```ts
// verifyTotp(secret, code, lastUsedStep) → { ok: boolean; step?: number }
// route 側: ok && step > user.totpLastUsedStep のときのみ成功、成功時に step を保存
```

### 3.3 リカバリーコード

**【現状】** SHA-256 単純ハッシュの JSON 配列（`users.mfaRecoveryCodes`）。コード自体が十分な乱数長であればハッシュ方式は妥当。

**【変更後】** 生成仕様を明文化する: `randomBytes(5)` × 2 を Base32 で `XXXXX-XXXXX` 形式・10 個。エントロピー 80bit のため SHA-256 直ハッシュを継続採用（scrypt 化は不要）。**残数が 3 個以下になったら `GET /api/auth/me` のレスポンスに `mfaRecoveryLow: true` を含めて再発行を促す**。使用時の監査ログ（既存 `mfa_recovery_used`）は維持。

## 4. CSRF・HTTP セキュリティ・レート制限

### 4.1 CSRF 防御（Origin / Sec-Fetch-Site 検証）

**【現状】** Cookie は `sameSite: lax` のみ。lax は**トップレベル GET ナビゲーションでは Cookie を送る**ため、GET で副作用を持たせない規律とセットでのみ成立している。トークン検証・Origin 検証は存在しない。

**【変更後】** `middleware.ts` に API 向けの検証を追加する（多層防御の 1 層目。Cookie 認証のブラウザリクエストのみが対象）。

```
対象: /api/* かつ method ∈ {POST, PUT, PATCH, DELETE}
免除: Authorization: Bearer 認証のリクエスト（モバイル。Cookie を使わないため CSRF 不成立）
判定:
  1. Sec-Fetch-Site ヘッダーがあれば same-origin / none 以外は 403
  2. Origin ヘッダーがあれば config.APP_ORIGIN（未設定時は Host から導出）と不一致なら 403
  3. どちらも無い場合（古いクライアント・curl）: Cookie 認証なら 403、Bearer なら通過
```

`POST /api/auth/login` も対象に含める（ログイン CSRF 対策）。`session-cleanup` コンテナは Bearer（サービスキー）のため影響しない。

## 5. 証憑ファイル（アップロード）の再設計

対象: `app/api/journals/[id]/receipts/route.ts`・`app/api/uploads/[filename]/route.ts`

### 5.1 配信側: 認証・テナント所有確認の必須化

**【現状】** `GET /api/uploads/[filename]` は **`requireRole` を一切呼んでおらず、無認証でアクセス可能**。ファイル名は UUID で推測困難だが、URL が漏れれば（監査ログ・ブラウザ履歴・Referer 経由）他テナントの証憑を誰でも取得できる。`path.basename()` によるトラバーサル対策のみ実装済み。

**【変更後】** 配信を Receipt レコード経由に変更し、認可を DB で判定する。

```

  2. receipt = db.receipt.findFirst({ where: { savedName: filename, journalEntry: { tenantId } } })
     （Receipt に savedName 列を追加。journalEntry リレーション経由でテナント境界を強制）
  3. 見つからなければ 404（他テナントのファイルも 404 に見える）

  5. レスポンスヘッダー:
     Content-Type: 保存時に検証済みの MIME（拡張子からの推測をやめ DB の値を使用）
     Content-Disposition: attachment; filename*=UTF-8''<encoded>   ← inline 表示による XSS 抑止
     X-Content-Type-Options: nosniff
     Cache-Control: private, no-store
```

### 5.2 アップロード側: 検証の追加

**【現状】** サイズ・拡張子・MIME の検証なし。`file.name.split(".").pop()` の拡張子をそのまま保存名に連結しており、`.html`・`.svg` 等も保存できる（配信は octet-stream になるが Content-Disposition が無いためブラウザ挙動に依存）。サイズ無制限はディスク枯渇 DoS になる。

**【変更後】** `lib/upload.ts`（新規）に検証を集約する。

| 検証 | 内容 |
|---|---|
| サイズ | `config.UPLOAD_MAX_BYTES`（既定 10MB）。超過は 413 |
| 拡張子ホワイトリスト | `pdf / jpg / jpeg / png / gif / webp` のみ（配信側 contentTypes マップと同一の定数を共有） |
| MIME 検証 | `file.type` 申告値と拡張子の整合 + **マジックバイト**（先頭バイト列: `%PDF`, `\xFF\xD8`, `\x89PNG`, `GIF8`, `RIFF..WEBP`）の 3 点一致。不一致は 400 |
| 保存名 | `randomUUID()` + 検証済み拡張子（ユーザー入力は保存パスに一切使わない。既存踏襲） |
| 保存先 | `uploads/<tenantId>/<uuid>.<ext>` にテナント別サブディレクトリ化。ディレクトリ分離により誤配信の影響範囲を限定 |
| メタデータ | `receipts.savedName`・`receipts.mimeType` 列を追加し、配信時の Content-Type は保存時検証値を使用 |
| 削除 | Receipt 削除 API（新設 `DELETE /api/journals/[id]/receipts/[receiptId]`、editor）でレコードと実ファイルを同時削除。仕訳削除時もカスケードでファイル削除 |

## 7. CSV / Excel 取込ロジックの再設計

対象: `lib/import.ts`・`lib/banktxn-import.ts`

**【現状】** `importRows()` は 1 行ごとに `findUnique` → `upsert` → `create` × 2 を直列実行（1,000 行で約 4,000 クエリ）。行数上限が無く、巨大 CSV の投稿でイベントループと DB を長時間占有できる。トランザクション外のため途中失敗で部分取込が残る。

|---|---|

### 9.1 ステップ 2: 配分ルールと予算一括登録

#### `lib/allocation.ts`（新規・純関数）

```ts
export type AllocationRule = { id: number; label: string; group: "fixed" | "living" | "other";
  minPercent: number; maxPercent: number | null; accountId: number | null; sortOrder: number };

// 基準額と控除額（ローン割賦）から科目別推奨額を算出する。I/O なし・Decimal 演算
export function suggestAllocation(input: {
  basisAmount: Decimal;                    // 収入合計（予算 or 実績）
  overlays: { accountId: number; amount: Decimal }[]; // ステップ 5 の割賦（控除対象）
  rules: AllocationRule[];
}): {
  available: Decimal;                      // 配分可能額 = basis − Σoverlay
  items: { rule: AllocationRule; min: Decimal; max: Decimal | null; recommended: Decimal }[];
  summary503020: { needs: Decimal; wants: Decimal; savings: Decimal };
}
```

- `recommended = available × (minPercent+maxPercent)/2 ÷ 100`（maxPercent null 時は min）。**銭単位で `Decimal.ROUND_HALF_UP`**、合計が available を超える場合は超過フラグを返し UI 側で警告。

| API | role | 設計 |
|---|---|---|

### 9.4 ステップ 5: 元利均等返済の自動計算

#### `lib/debt-schedule.ts` への追加（純関数）

```ts
// 元利均等の月額。r=0（無利子）は P/n の単純割り。丸めは銭→円で ROUND_HALF_UP
export function annuityMonthlyPayment(principal: Decimal, annualRatePercent: Decimal, months: number): Decimal;

// 月別償還スケジュール。最終回で丸め誤差を清算し、Σ元本 = principal を厳密に保証する
export function amortizationSchedule(principal: Decimal, annualRatePercent: Decimal,
  startMonth: YearMonth, endMonth: YearMonth):
  { month: YearMonth; payment: Decimal; principal: Decimal; interest: Decimal; remaining: Decimal }[];
```

- **Decimal 演算を必須**とする（`(1+r)^n` は Decimal.pow）。浮動小数の混入を防ぐため、引数は `Decimal` 型のみ受け付ける。
- 検証: `months ≥ 1`・`annualRatePercent ∈ [0, 100]`・`principal > 0` を関数冒頭で assert（Zod は API 層、純関数は事前条件チェックの二重化）。
- `GET /api/budgets` のオーバーレイ: 既存 `housingLoanOverlay` を `loanOverlay` に改称し、`linkedAccountId` を持つ**全ローン**に拡大。`monthlyPayment` 入力済みなら入力値優先、未入力なら `annuityMonthlyPayment` の計算値（仕様書 §6.5 どおり）。オーバーレイ額の由来（`source: "manual" | "annuity"`）をレスポンスに含め、UI のバッジ表示と監査可能性を確保する。
- 残高推移グラフは `amortizationSchedule` の `remaining` 系列に置換（現状の線形補間を廃止）。実返済（`loan_repayments`）がある月は実績値を優先し、以降のスケジュールは残高から再計算する。

### 9.5 ステップ 6: 表示名解決と学習ガイド
